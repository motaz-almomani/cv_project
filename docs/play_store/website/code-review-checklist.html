Firebase & security practices

Firestore Security Rules

Critical: This repo does not include committed firestore.rules. You must deploy rules so that only authenticated users read/write documents they own — for example cvs documents where request.auth.uid == resource.data.userId on read/update/delete and the same check on create. Without rules (or with defaults), data can be exposed.
  • Deny all by default; allow cvs CRUD only when userId matches request.auth.uid.
  • Review indexes if you use compound queries; your app queries cvs where userId == uid (may need a composite index at scale).
  • Enable Firebase App Check for Android once rules are stable, to reduce abuse from non-app clients.

Firebase Authentication

  • Email/password is implemented via AuthService; errors are surfaced user-friendly.
  • Strength: Password changes use reauthenticateWithCredential — good practice.
  • Consider enforcing email verification for sensitive flows; the app exposes sendEmailVerification — ensure product copy explains verification status.
  • No in-app account deletion in current code paths — aligns with documenting email-based deletion, but Play increasingly expects accessible account deletion UX; planning an in-app path is recommended.

Secrets & configuration

  • google-services.json / firebase_options.dart contain public client identifiers — acceptable in client apps; protect production API keys for any callable backend separately.
  • cloud_functions is listed in pubspec.yaml but unused in Dart grep — remove the dependency if you are not deploying functions, to shrink attack surface and dependencies.

Common Google Play policy risks (for this app type)

  • Data safety: Declarations must match actual SDK/data flows (Auth, Firestore, any future Analytics/Crashlytics).
  • Account deletion: Provide a discoverable process (URL + in-app path when possible) consistent with Play’s account deletion requirements.
  • Permissions: Only request permissions you use; document storage if you add file pickers or exports.
  • Deceptive behaviour / metadata: Store listing, screenshots, and “AI” claims must match functionality (current analysis appears to be local scoring, not a remote model — keep listing accurate).
